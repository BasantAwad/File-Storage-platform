Distributed Systems project specification.
1. Project Objective
Students will design and implement a distributed, production-style file storage platform composed of 40 independent microservices.
The system simulates:
Chunk-based distributed storage
Replication and re-replication
Failure detection and recovery
Distributed metadata coordination
Observability
Configuration management
Chaos engineering
CI/CD automation
Kubernetes orchestration
Workflow automation using n8n (bonus)
The final system must behave like a simplified cloud object storage system.
2. Global Technical Rules
Each service MUST:
Be fully independent
Own its database (no shared DB)
Communicate via REST (synchronous) and Apache Kafka (asynchronous event streaming)
Expose:
GET /health
GET /ready
Follow standard JSON response format
Be containerized
Have Kubernetes deployment files
Be deployed with replicas = 2
Include tests
Include API documentation
2.1 Service Design Deliverables
For EVERY microservice, you must submit a short design package that includes:
• Data Ownership:     Database schema, indexes, constraints, and the data lifecycle managed by the service (ERD).
Architecture Diagram (Event Driven): A diagram showing where the service fits within the overall system architecture and how it interacts with other services.
 Sequence Diagram: for each microservices illustrating the main workflow of the service
Collaboration Diagram:A diagram showing how system components collaborate and exchange messages to accomplish a specific task.
High-Level Block Diagram:A high-level diagram that illustrates the main system components and their relationships
Deployment Diagram:A diagram illustrating how services are deployed in the system infrastructure (containers, nodes, servers, or cloud components).
Each diagram must include at least two references notation to real-world systems or architectures (e.g., AWS, Google Cloud, Netflix) that implement similar architectural patterns.
2.2 Service Communication Classification
Every service is classified into one of three communication patterns based on the following decision rule: 
(1) If a user or external system calls the service and waits for an immediate answer, and no other service needs to react to the result — use Pure REST. 
(2) If a user calls the service and waits for an answer, AND other services must react to what happened — use Hybrid REST + Kafka.
 (3) If no user calls the service directly and it only reacts to events from other services — use Pure Kafka Consumer.

Category 1 — Pure REST Only (7 Services)
These services respond synchronously and no other service needs to react to their result. Adding Kafka would bring zero benefit.
#
Service
Why Pure REST Only
6
Rate Limit
Real-time enforcement — must block or allow the request instantly. By the time a Kafka event is published and consumed, the request has already passed. Adding Kafka adds latency with zero benefit.
9
Chunk Catalog
Upload Session already publishes upload.completed which carries all chunk IDs. Publishing a separate Kafka event per chunk would create thousands of messages per upload with no additional consumers.
10
Chunk Location
Location tracking is a lookup and update operation. Replication events already handle location changes downstream. No other service needs to react to a location registration in real time.
14
Tagging
Tag operations are low-importance metadata changes. Search Index already re-indexes on upload.completed. No service reacts meaningfully to tag changes in real time.
15
Comments
Pure user-facing CRUD. No other service reacts to a new comment. Audit logging can be done via a direct lightweight REST call at negligible cost.
33
Secrets Broker
SECURITY CONSTRAINT — publishing secrets or secret-related events to Kafka is a security risk. Kafka is not encrypted by default. Secrets must stay in private synchronous channels only.
36
Presigned URL
URL generation is a pure synchronous operation. The URL itself is the response — client needs it immediately to start uploading or downloading. Nothing else reacts to URL creation.

No user-facing REST endpoint triggers their core work. They run permanently as background workers. If they go down, Kafka holds all messages and they catch up automatically on restart.

  "success": true,
  "data": {},
  "meta": {
    "service": "service-name",
    "request_id": "uuid"
  }
}
Error:
{
  "success": false,
  "error": {
    "code": "ERROR_CODE",
    "message": "Description",
    "details": {}
  },
  "meta": {
    "service": "service-name",
    "request_id": "uuid"
  }
}
